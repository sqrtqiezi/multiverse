## 1. 概述

### 1.1 目标

为 Multiverse CLI 实现统一的错误处理机制，当出错时向用户提供清晰的错误信息和解决建议。

### 1.2 用户故事

作为开发者，当出错时我希望看到清晰的错误信息。

### 1.3 验收标准

- Docker 未安装时给出友好提示
- 凭证未找到时给出登录指引
- 容器启动失败时显示详细错误
- 文件系统错误时给出权限和空间检查建议
- 所有错误都有解决建议

## 2. 错误分类

### 2.1 错误码定义

```typescript
enum ErrorCode {
  // Docker 相关
  DOCKER_NOT_AVAILABLE = 'DOCKER_NOT_AVAILABLE',
  DOCKER_PERMISSION_DENIED = 'DOCKER_PERMISSION_DENIED',
  
  // 凭证相关
  CREDENTIALS_NOT_FOUND = 'CREDENTIALS_NOT_FOUND',
  CREDENTIALS_INVALID = 'CREDENTIALS_INVALID',
  
  // 容器相关
  CONTAINER_START_FAILED = 'CONTAINER_START_FAILED',
  IMAGE_PULL_FAILED = 'IMAGE_PULL_FAILED',
  
  // 文件系统相关
  WORKSPACE_NOT_WRITABLE = 'WORKSPACE_NOT_WRITABLE',
  DISK_SPACE_INSUFFICIENT = 'DISK_SPACE_INSUFFICIENT',
  VERSE_FILE_CORRUPTED = 'VERSE_FILE_CORRUPTED',
  
  // 通用错误
  UNKNOWN_ERROR = 'UNKNOWN_ERROR',
}
```

### 2.2 错误场景说明

| 错误码 | 触发场景 | 检测时机 |
|--------|---------|---------|
| DOCKER_NOT_AVAILABLE | Docker daemon 未运行或未安装 | 启动前 |
| DOCKER_PERMISSION_DENIED | 当前用户无 Docker 权限 | 启动前 |
| CREDENTIALS_NOT_FOUND | `~/.claude/` 目录不存在 | 启动前 |
| CREDENTIALS_INVALID | 凭证文件格式错误或已过期 | 启动前 |
| CONTAINER_START_FAILED | 容器启动失败（通用） | 运行时 |
| IMAGE_PULL_FAILED | Docker 镜像拉取失败 | 运行时 |
| WORKSPACE_NOT_WRITABLE | `.multiverse/` 目录无写权限 | 启动前 |
| DISK_SPACE_INSUFFICIENT | 磁盘空间不足 | 启动前 |
| VERSE_FILE_CORRUPTED | Verse 文件损坏无法读取 | 运行时 |
| UNKNOWN_ERROR | 未分类的错误（兜底） | 任意时机 |

### 3.1 核心组件

```

               │ try-catch
               ▼
┌─────────────────────────────────────────┐
│          ErrorHandler                   │
│  (packages/core/src/error-handler.ts)   │
│  • format(error): FormattedError        │
│  • 错误码 → 友好信息                     │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│       Error Templates                   │
│  • 每个错误码的信息模板                  │
│  • 支持变量替换                          │
└─────────────────────────────────────────┘
```

### 3.2 数据结构

```typescript
// 应用错误
interface AppError {
  code: ErrorCode;
  cause?: Error;                    // 原始错误
  context?: Record<string, any>;    // 上下文信息
}

// 格式化后的错误
interface FormattedError {
  title: string;                    // 问题标题
  description: string;              // 问题描述
  reason?: string;                  // 原因分析
  suggestions: string[];            // 解决建议（多条）
  exitCode: number;                 // 进程退出码
}

// 错误处理器接口
interface IErrorHandler {
  format(error: AppError): FormattedError;
}
```

### 3.3 错误信息模板

每个错误码对应一个模板，包含：
- **title**: 简短的问题标题
- **description**: 详细的问题描述
- **reason**: 可能的原因分析（可选）
- **suggestions**: 解决建议列表
- **exitCode**: 进程退出码

**模板示例**：

## 4. 错误检测机制

### 4.1 启动前检测

在容器启动前执行以下检测，快速失败：

```typescript
async function preflightChecks(): Promise<void> {
  // 1. Docker 可用性
  await checkDockerAvailable();
  
  // 2. Docker 权限
  await checkDockerPermission();
  
  // 3. 凭证文件
  await checkCredentials();
  
  // 4. 工作区权限
  await checkWorkspaceWritable();
  
  // 5. 磁盘空间
  await checkDiskSpace();
}
```

**检测逻辑**：
- **Docker 可用性**: 执行 `docker info`，检查返回码
- **Docker 权限**: 检查错误信息是否包含 "permission denied"
- **凭证文件**: 检查 `~/.claude/` 目录是否存在
- **工作区权限**: 尝试在 `.multiverse/` 创建临时文件
- **磁盘空间**: 检查可用空间是否 > 100MB

### 4.2 运行时检测

容器启动过程中的错误检测：

```typescript
async function startContainer(): Promise<void> {
  try {
    // 拉取镜像
    await pullImage();
  } catch (error) {
    throw new AppError({
      code: ErrorCode.IMAGE_PULL_FAILED,
      cause: error,
    });
  }
  
  try {
    // 启动容器
    await dockerClient.start();
  } catch (error) {
    throw new AppError({
      code: ErrorCode.CONTAINER_START_FAILED,
      cause: error,
    });
  }
}
```

## 5. CLI 集成

### 5.1 入口错误捕获

```typescript
// packages/cli/src/index.ts
import { errorHandler } from '@multiverse/core';

async function main() {
  try {
    await program.parseAsync(process.argv);
  } catch (error) {
    const appError = toAppError(error);
    const formatted = errorHandler.format(appError);
    
    console.error(formatErrorOutput(formatted));
    process.exit(formatted.exitCode);
  }
}

main();
```

### 5.2 错误输出格式

使用颜色和图标增强可读性：

```
❌ Docker 不可用

Docker daemon 未运行或未安装。

💡 解决建议：
  1. 检查 Docker 是否已安装：docker --version
  2. 启动 Docker daemon：sudo systemctl start docker
  3. 确认当前用户在 docker 组：sudo usermod -aG docker $USER
```

**格式化函数**：
```typescript
function formatErrorOutput(error: FormattedError): string {
  const lines: string[] = [];
  
  // 标题（红色）
  lines.push(chalk.red(`❌ ${error.title}`));
  lines.push('');
  
  // 描述
  lines.push(error.description);
  
  // 原因（如果有）
  if (error.reason) {
    lines.push('');
    lines.push(chalk.dim(`原因：${error.reason}`));
  }
  
  // 建议（黄色）
  if (error.suggestions.length > 0) {
    lines.push('');
    lines.push(chalk.yellow('💡 解决建议：'));
    error.suggestions.forEach((suggestion, index) => {
      lines.push(chalk.yellow(`  ${index + 1}. ${suggestion}`));
    });
  }
  
  return lines.join('\n');
}
```

### 6.2 单元测试（Vitest）

测试 ErrorHandler 的各种场景：

```typescript
describe('ErrorHandler', () => {
  it('should format DOCKER_NOT_AVAILABLE error', () => {
    const error: AppError = {
      code: ErrorCode.DOCKER_NOT_AVAILABLE,
    };
    
    const formatted = errorHandler.format(error);
    
    expect(formatted.title).toBe('Docker 不可用');
    expect(formatted.suggestions).toHaveLength(3);
    expect(formatted.exitCode).toBe(1);
  });
  
  it('should replace variables in error message', () => {
    const error: AppError = {
      code: ErrorCode.WORKSPACE_NOT_WRITABLE,
      context: { path: '/home/user/project/.multiverse' },
    };
    
    const formatted = errorHandler.format(error);
    
    expect(formatted.description).toContain('/home/user/project/.multiverse');
  });
  
  it('should handle unknown error code', () => {
    const error: AppError = {
      code: 'INVALID_CODE' as ErrorCode,
    };
    
    const formatted = errorHandler.format(error);
    
    expect(formatted.code).toBe(ErrorCode.UNKNOWN_ERROR);
  });
});
```

## 8. 非功能性需求

### 8.1 性能

- 启动前检测总耗时 < 500ms
- 错误格式化耗时 < 10ms

### 8.2 可维护性

- 新增错误类型只需添加错误码和模板
- 错误信息支持国际化（预留接口）

### 8.3 用户体验

- 错误信息清晰易懂
- 解决建议可操作
- 输出格式美观（颜色、图标）

## 9. 风险与应对

### 9.1 风险：检测逻辑误判

**场景**：Docker 实际可用，但检测失败

**应对**：
- 提供 `--skip-checks` 参数跳过检测
- 记录详细日志便于排查

### 9.2 风险：错误信息不准确

**场景**：错误原因复杂，建议不适用

**应对**：
- 提供通用建议 + 原始错误信息
- 引导用户查看日志或提交 issue

## 10. 总结

### 10.1 设计要点

1. **集中式错误处理** - 统一的 ErrorHandler 管理所有错误
2. **友好的错误信息** - 标题 + 描述 + 原因 + 建议
3. **启动前检测** - 快速失败，避免无效操作
4. **可扩展架构** - 易于添加新错误类型

### 10.2 验收标准映射

| 验收标准 | 实现方式 |
|---------|---------|
| Docker 未安装时给出友好提示 | DOCKER_NOT_AVAILABLE 错误码 + 模板 |
| 凭证未找到时给出登录指引 | CREDENTIALS_NOT_FOUND 错误码 + 模板 |
| 容器启动失败时显示详细错误 | CONTAINER_START_FAILED 错误码 + 原始错误 |
| 文件系统错误时给出建议 | WORKSPACE_NOT_WRITABLE、DISK_SPACE_INSUFFICIENT |
| 所有错误都有解决建议 | 每个模板都包含 suggestions 字段 |
