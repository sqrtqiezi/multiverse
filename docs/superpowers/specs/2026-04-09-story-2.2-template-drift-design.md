# Story 2.2 Design: Template Drift Detection And User-Driven Sync

## 1. 背景与目标

Story 2.2 的原始目标是“配置文件版本管理”，希望在全局配置变化后，系统能够帮助用户识别当前配置与已绑定 template 是否一致，并在需要时同步到新的 template。

结合当前项目阶段和产品诉求，本设计不引入常驻 watcher 或后台 daemon，而是采用“命令触发的机会性检测”：

- 每次执行 `start` 或后续 `verse` 相关命令时，快速检查当前全局配置是否偏离 verse 绑定的 template
- 如果已偏离，由用户决定是否基于当前最新配置创建一个新的 template
- template 保持不可变，不支持覆盖原 template

本设计覆盖以下目标：
- 新建 template 时始终使用最新的磁盘配置
- `start` 能发现当前配置与已绑定 template 的漂移
- 用户可选择继续使用旧 template，或创建新 template 并切换当前 verse
- 核心检测与同步逻辑沉入 `core`，供 CLI 和未来 GUI 复用

## 2. 范围与非范围

范围：
- 为 template 增加稳定内容指纹（fingerprint）
- 在 core 中实现当前配置与 template 快照的漂移检测
- 在 CLI `start` 命令中接入漂移提示与交互式同步
- 为后续 `verse` 命令预留复用的检测接口
- 新增 Story 2.2 的 E2E Gherkin 测试和对应单元测试

非范围：
- 常驻 watcher / daemon / 文件系统实时监听
- 完整配置历史时间线或事件溯源仓库
- template 原地覆盖更新
- GUI 页面实现
- 完整逐行 diff 展示

## 3. 核心语义

### 3.1 配置对象范围

Story 2.2 的配置对象为完整全局配置集：
- `CLAUDE.md`
- `~/.claude/**`

这里的语义与 Story 2.1 的 template 快照保持一致，避免 template、drift check、未来 GUI 对“配置”含义不一致。

### 3.2 漂移（Drift）的定义

若当前磁盘上的完整配置集与 verse 当前绑定 template 的快照内容不一致，则认为发生了 drift。

判断标准基于稳定指纹，而不是文件系统事件：
- 对 template 快照计算规范化指纹
- 对当前磁盘配置重新扫描并计算同样规则的指纹
- 指纹不同即视为 drift

### 3.3 用户驱动同步

检测到 drift 后，不自动改写 template，也不自动更新 verse。

用户可执行的动作仅保留三种：
- `keep`：继续使用旧 template 执行当前命令
- `sync-and-switch`：基于当前配置创建新 template，并把当前 verse 切换到新 template
- `cancel`：终止当前命令

设计约束：
- template 不可变
- 同步总是创建一个新 template
- verse 始终只绑定一个 templateId，不维护独立版本链

## 4. 数据模型

### 4.1 Template 扩展

在现有 `Template` 结构上新增 `fingerprint` 字段：

```ts
interface Template {
  id: string;
  name: string;
  description?: string;
  snapshot: ConfigSnapshot;
  fingerprint: string;
  createdAt: string;
}
```

语义：
- `fingerprint` 表示该 template 快照的稳定内容标识
- 创建 template 时计算并持久化
- 已存在但没有 `fingerprint` 的历史 template，首次读取时允许现场补算

### 4.2 Drift Check 结果

新增用于 CLI / GUI 复用的结果结构：

```ts
interface ConfigDriftCheckResult {
  templateId: string;
  templateName: string;
  templateFingerprint: string;
  currentFingerprint: string;
  isDrifted: boolean;
  addedFiles: string[];
  modifiedFiles: string[];
  removedFiles: string[];
}
```

约束：
- 文件路径均使用相对配置根的逻辑路径
- 三类列表按字典序排序
- 结果既可用于交互提示，也可用于后续 GUI 状态展示

### 4.3 Verse 关系

本设计不改变 verse 的主模型，只继续依赖已有 `templateId`：

```ts
interface Verse {
  // existing fields...
  templateId: string;
}
```

语义：
- verse 指向当前 branch 使用中的 template
- 当用户选择 `sync-and-switch` 时，仅更新 `templateId`
- 不在 2.2 中引入 `configVersionId` 或历史绑定记录

## 5. 指纹与规范化规则

`fingerprint` 必须对相同配置内容稳定，对无关因素不敏感。

建议规则：
- 路径排序后参与哈希
- `CLAUDE.md` 作为固定逻辑路径处理
- `.claude/**` 使用相对路径
- 文本内容统一为 UTF-8
- 换行统一为 `\n`
- 路径分隔符统一为 `/`

指纹输入建议采用“规范化清单”形式：

```text
CLAUDE.md\n<content>\n
.claude/settings.json\n<content>\n
.claude/commands/foo.md\n<content>\n
```

再对该规范化文本做 SHA-256。

2.2 不做 JSON 语义归一化。也就是说：
- `settings.json` 的内容按文件原文参与哈希
- 如果用户仅调整空格或字段顺序，仍视为变化

理由：
- 行为简单明确
- 与 template 快照“按文件内容冻结”的语义一致
- 避免在 2.2 引入 JSON 解析失败、重排规则、语义 diff 等额外复杂度

## 6. 组件设计

### 6.1 `template-fingerprint.ts`

职责：对 template 快照或当前配置生成稳定指纹。

建议接口：
- `computeSnapshotFingerprint(snapshot: ConfigSnapshot): string`
- `computeCurrentConfigFingerprint(homeDir: string): Promise<{ fingerprint: string; snapshot: ConfigSnapshot }>`

说明：
- 第二个接口可直接复用现有 `createConfigSnapshot`
- 返回 snapshot 是为了避免 drift 检测后、同步创建 template 时再次重复扫描

### 6.2 `config-drift.ts`

职责：比较当前配置与目标 template 的差异，生成用户可消费的结果。

建议接口：
- `checkTemplateDrift(params): Promise<ConfigDriftCheckResult>`

输入建议包含：
- `homeDir`
- `template`

输出语义：
- 比较 template snapshot 和当前 snapshot
- 生成 `addedFiles` / `modifiedFiles` / `removedFiles`
- 基于 fingerprint 给出 `isDrifted`

### 6.3 `TemplateService`

职责扩展：
- 创建 template 时写入 `fingerprint`
- 提供基于“当前磁盘配置”创建新 template 的统一入口

建议新增/调整接口：
- `create(...)`：内部自动计算 fingerprint
- `createFromCurrentConfig(...)`：与 drift sync 共享同一创建路径

命名规则建议：
- 用户显式创建时仍使用用户给定名称
- drift sync 创建新 template 时，采用可预测的派生名称，例如 `<old-name>-sync-<timestamp>`
- 若名称冲突，追加短后缀重试

### 6.4 `VerseService`

职责扩展：
- 暴露安全的 template 切换接口

建议接口：
- `updateTemplateForCurrentBranch(cwd: string, templateId: string): Promise<void>`

约束：
- 只更新当前 branch 对应 verse
- 保持现有 atomic write / lock 语义
- 不允许 CLI 直接绕过 service 修改 verse 文件

## 7. CLI 流程设计

### 7.1 `start` 接入点

`packages/cli/src/commands/start.ts` 在确定当前 verse 和 template 后执行 drift check：

1. 解析当前 branch 对应 verse
2. 如果没有 verse 或 verse 没有关联 template，则跳过 drift 检测
3. 加载 verse 对应 template
4. 执行 `checkTemplateDrift`
5. 若 `isDrifted = false`，继续启动
6. 若 `isDrifted = true`，输出摘要并进入交互

### 7.2 交互动作

CLI 提示示例：

```text
Current global config has drifted from template "default".
Modified: 2, Added: 1, Removed: 0
Examples: CLAUDE.md, .claude/settings.json, .claude/commands/review.md

Choose an action:
1. Keep using current template
2. Create a new template from current config and switch this verse
3. Cancel
```

动作映射：
- 选 `1`：继续当前 `start`
- 选 `2`：
  - 基于当前配置创建新 template
  - 更新当前 verse 的 `templateId`
  - 输出新 template 名称和 id
  - 继续当前 `start`
- 选 `3`：退出且不修改任何状态

### 7.3 后续 `verse` 命令复用

虽然 2.2 不要求立即实现独立 `verse` 管理命令，但 core 逻辑必须是可复用的。

后续 `verse` 命令可使用同样的流程：
- 查询 verse 时显示 drift 状态
- 手动触发同步时调用同一套 `createFromCurrentConfig + updateTemplateForCurrentBranch`

## 8. 变更摘要策略

2.2 不实现完整文本 diff，仅提供足够做决策的摘要。

摘要生成规则：
- 基于逻辑路径集合比较
- 路径只分三类：新增、删除、修改
- CLI 中显示计数
- 最多显示前 5 个变更路径作为示例

判断规则：
- path 在当前快照存在、template 快照不存在 -> `added`
- path 在 template 快照存在、当前快照不存在 -> `removed`
- path 同时存在但内容不同 -> `modified`

这足以支持用户判断“要不要同步”，也方便 GUI 以后做列表展示。

## 9. 错误处理

新增错误场景与处理原则：

1. 当前配置读取失败
- 终止当前命令
- 明确输出失败文件路径和系统错误

2. verse 绑定的 template 不存在或损坏
- 终止当前命令
- 提示当前 verse 的 template 已不可读取，需要手动重新同步

3. `sync-and-switch` 中 template 创建失败
- 终止当前命令
- 不隐式退化为 `keep`

4. verse 更新失败
- 终止当前命令
- 已创建的新 template 保留，提示用户可后续手动切换

5. 历史 template 缺少 `fingerprint`
- 首次读取时现场补算
- 不要求一次性批量迁移

错误原则：
- 不静默跳过漂移
- 不自动做用户未确认的状态写入
- 提示必须能指导用户下一步动作

## 10. 测试策略（遵循先 E2E 后单测）

### 10.1 E2E（Gherkin）

建议新增 `e2e/features/template-drift.feature`，至少覆盖以下场景：

1. `start` 时配置未变化，不提示同步
- Given 当前 branch 已绑定 template
- And 当前全局配置与 template 一致
- When 执行 `multiverse start`
- Then 启动继续执行
- And 不出现 drift 提示

2. `start` 时检测到 `CLAUDE.md` 变化
- Given 当前 branch 已绑定 template
- And 用户修改了 `CLAUDE.md`
- When 执行 `multiverse start`
- Then 出现 drift 提示

3. 用户选择 `keep`
- Given 检测到 drift
- When 用户选择 keep
- Then 当前命令继续
- And verse 仍绑定旧 template

4. 用户选择 `sync-and-switch`
- Given 检测到 drift
- When 用户选择 sync-and-switch
- Then 创建一个新的 template
- And 当前 verse 切换到新 template
- And 当前命令继续

5. 用户选择 `cancel`
- Given 检测到 drift
- When 用户选择 cancel
- Then 当前命令终止
- And 不创建新 template
- And verse 保持不变

### 10.2 单元测试（core）

`template-fingerprint.test.ts`：
- 相同快照生成相同 fingerprint
- 文件顺序变化不影响 fingerprint
- 换行规范化生效
- 内容变化导致 fingerprint 变化

`config-drift.test.ts`：
- 无变化时 `isDrifted = false`
- 新增文件被识别
- 删除文件被识别
- 修改文件被识别
- 结果按路径排序

`template-service.test.ts`：
- 创建 template 时持久化 fingerprint
- drift sync 创建新 template 使用当前磁盘配置
- 派生名称冲突时正确避让

`verse-service.test.ts`：
- 可更新当前 branch 的 templateId
- 仅更新目标 verse，不破坏其他字段

### 10.3 CLI 测试

`start-command.test.ts`：
- no-drift 时不进入交互
- drift + keep 时继续执行
- drift + sync-and-switch 时创建新 template 并更新 verse
- drift + cancel 时中止
- drift 检测或同步失败时输出明确错误

## 11. 实施边界与后续演进

本设计刻意不把 2.2 做成完整“配置版本系统”，原因如下：
- 当前没有 daemon 架构，不值得为了版本完整性引入常驻服务
- 用户更关心“当前配置有没有偏离已绑定 template”以及“是否要同步”
- template 已经是成熟的不可变快照模型，直接在它上面增加 drift 语义更自然

如果后续需要更强能力，可在后续 story 上继续演进：
- 显式 `multiverse verse status` / `multiverse verse sync`
- GUI 中显示 drift 状态和文件摘要
- 手动查看更详细 diff
- 引入真正的配置历史时间线

## 12. 验收标准映射

本设计将 Story 2.2 的原始验收目标重解释为以下可交付行为：

- 修改配置后，系统能在后续命令执行时检测到变化
- 系统可以识别当前 verse 绑定 template 是否已过期
- 用户可以显式创建新的 template 来承载最新配置
- 用户可以将当前 verse 切换到新的 template

这满足当前产品阶段“把主动权交给使用者”的要求，同时避免 watcher 带来的系统负担和实现复杂度。
