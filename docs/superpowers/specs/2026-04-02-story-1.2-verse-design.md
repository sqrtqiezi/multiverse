# Story 1.2 Design: 自动创建 Verse 与 Run 记录

## 1. 背景与目标

Story 1.2 目标：在 `multiverse start` 流程中自动记录运行历史。

本设计覆盖以下验收标准：
- 首次运行自动创建 verse
- Verse 关联当前 Git branch
- 退出后自动保存 run 记录
- 可以直接查看 verse JSON 文件

约束：
- 存储位置使用项目内路径，不使用全局用户目录
- 数据格式采用单文件 JSON（每个 branch 一个文件）
- 在并发和异常场景下，优先保证文件一致性与可恢复性

## 2. 范围与非范围

范围：
- 在 core 中新增 Git branch 解析、Verse 仓储和 Verse 服务
- 在 CLI `start` 命令中接入 run 开始/结束记录
- 新增 Story 1.2 的 E2E Gherkin 测试和对应单元测试

非范围：
- verse 归档/fork 关系
- template 匹配与 branch 迁移提示
- 多进程分布式锁（仅本机文件锁语义）
- 复杂分析字段（duration、status、errorMessage 等）

## 3. 存储模型

### 3.1 路径与命名

- 根目录：`<projectRoot>/.multiverse/verses/`
- 文件名：`<sanitized-branch>.json`
- branch 名清洗规则：
  - 保留字符集：`[a-zA-Z0-9._-]`
  - 其他字符全部替换为 `_`

示例：
- `main` -> `main.json`
- `feature/story-1.2` -> `feature_story-1.2.json`

### 3.2 数据结构

```ts
interface RunRecord {
  runId: string;
  startAt: string; // ISO 8601
  endAt?: string; // ISO 8601
  exitCode?: number;
  containerId?: string;
}

interface Verse {
  id: string;
  branch: string;
  createdAt: string; // ISO 8601
  updatedAt: string; // ISO 8601
  runs: RunRecord[];
}
```

设计约束：
- `runs` 按时间追加，不重排
- `updatedAt` 在任何 run 变更时更新
- 不删除历史 run

## 4. 组件设计

### 4.1 `BranchResolver`（core）

职责：解析当前 Git branch。

接口：
- `getCurrentBranch(cwd: string): Promise<string>`

行为：
- 优先执行 `git rev-parse --abbrev-ref HEAD`
- 结果为 `HEAD`（detached）时返回 `detached-head`
- Git 不可用或非仓库目录时报错（CLI 统一处理）

### 4.2 `VerseRepository`（core）

职责：文件系统读写、加锁、原子写入、最小 schema 校验。

接口：
- `loadOrInit(params): Promise<Verse>`
- `saveAtomic(params): Promise<void>`
- `withLock(params, fn): Promise<T>`

内部行为：
- 目录不存在时自动创建 `.multiverse/verses`
- 锁文件：`<verseFile>.lock`
- 临时文件：`<verseFile>.tmp-<pid>-<timestamp>`

### 4.3 `VerseService`（core）

职责：实现业务语义，不直接暴露底层文件细节。

接口：
- `ensureVerseForCurrentBranch(cwd): Promise<{ verse, versePath }>`
- `appendRunStart(cwd, runId, startAt): Promise<void>`
- `finalizeRun(cwd, runId, endAt, exitCode, containerId?): Promise<void>`

语义：
- `appendRunStart`：创建一条进行中 run（仅 `runId/startAt`）
- `finalizeRun`：按 `runId` 更新结束字段
- `finalizeRun` 找不到 `runId` 时抛业务错误，避免静默数据丢失

## 5. JSON 写入流程（关键）

目标：在并发、崩溃、中断情况下，不损坏 verse 文件。

### 5.1 并发控制

1. 计算路径：`versePath`、`lockPath`、`tmpPath`
2. 尝试 `open(lockPath, 'wx')` 获取独占锁
3. 若锁存在：指数退避重试（50ms, 100ms, 200ms, 400ms, ...）
4. 总等待超过 `3000ms`，抛 `VerseLockTimeoutError`
5. 策略：超时即失败并提示重试（不抢锁）

### 5.2 读取与校验

持锁状态下执行：
1. 如果 `versePath` 不存在，初始化新 verse
2. 如果存在，读取并 `JSON.parse`
3. 执行最小 schema 校验：
   - 顶层必须包含 `id/branch/createdAt/updatedAt/runs`
   - `runs` 必须是数组
4. 校验失败抛 `VerseCorruptedError`，且不覆盖原文件

### 5.3 变更与提交

1. 在内存对象上执行业务变更（append/finalize）
2. 更新 `updatedAt`
3. `writeFile(tmpPath, serializedJson)`
4. `fsync(tmpFd)`（实现允许在不支持场景降级）
5. `rename(tmpPath, versePath)` 完成原子替换
6. `finally` 清理残留 `tmpPath`

### 5.4 释放锁

`finally` 中始终执行：
- `close(lockFd)`
- 删除 `lockPath`

任何异常都不能跳过锁释放。

### 5.5 启动时恢复

`ensureVerseForCurrentBranch` 前置执行轻量恢复：
- 清理同目录下超时残留 `*.tmp-*`（如 >24h）
- 对陈旧 `.lock`（mtime 过旧）仅记录 warning，不自动抢锁

备注：锁回收策略保持保守，避免误删活跃写入锁。

## 6. CLI 接入流程

`packages/cli/src/commands/start.ts` 流程扩展：

1. 通过 `VerseService` 确保当前 branch verse 存在
2. 生成 `runId`，调用 `appendRunStart`
3. 执行已有容器启动与 attach 流程
4. 容器退出后调用 `finalizeRun(runId, endAt, exitCode, containerId)`
5. 若容器已创建但流程异常，也尝试 `finalizeRun(..., exitCode=1)`
6. 若容器未成功创建，不写 run 结束记录

控制台输出（新增）：
- `✓ Verse ready: .multiverse/verses/<file>.json`
- `✓ Run recorded: <runId>`（结束后）

## 7. 错误处理

新增错误类型：
- `VerseLockTimeoutError`
- `VerseCorruptedError`
- `RunNotFoundError`

CLI 映射策略：
- 锁超时：提示“同一 branch 可能有并发写入，请稍后重试”
- 文件损坏：提示文件路径并建议手动备份后修复
- run 不存在：提示内部状态不一致并输出 runId

错误原则：
- 明确错误原因
- 包含可操作建议
- 不吞异常

## 8. 测试策略（遵循先 E2E 后单测）

## 8.1 E2E（Gherkin）

新增 `e2e/features/verse.feature`：

1. `首次 start 自动创建 verse 文件`
- Given 在 git branch `main`
- When 执行 `multiverse start`
- Then 存在 `.multiverse/verses/main.json`
- And verse 中 `runs.length = 1`

2. `再次 start 追加 run`
- Given 已存在当前 branch verse
- When 再次执行 `multiverse start`
- Then `runs.length` 递增

3. `容器退出后补齐 run 结束字段`
- When run 结束
- Then 最新 run 包含 `endAt/exitCode/containerId`

说明：Docker 依赖场景沿用现有 `MULTIVERSE_E2E_DOCKER=1` 门控。

## 10. 风险与缓解

风险：并发 `start` 导致同一 verse 竞争写入。  
缓解：文件锁 + 超时失败策略，避免破坏数据。

风险：异常中断导致 run 只有 start 无 finish。  
缓解：允许该状态存在，后续可作为“异常退出”依据。

风险：JSON 人工编辑损坏。  
缓解：schema 校验 + 拒绝覆盖 + 错误提示。

风险：单 branch 长期累积导致 `<sanitized-branch>.json` 文件持续膨胀，读取和写入延迟上升。  
缓解：Story 1.2 先保持单文件实现；后续版本引入滚动归档与主文件瘦身策略（见第 12 节）。

## 11. 验收映射

- 首次运行自动创建 verse：通过 `ensureVerseForCurrentBranch`
- Verse 关联当前 branch：通过 `BranchResolver` + 文件命名
- 退出后保存 run：通过 `appendRunStart/finalizeRun`
- 可查看 JSON 文件：落盘到 `.multiverse/verses/*.json`

## 12. 后续演进（Story 1.2 非范围）

### 12.1 问题陈述

随着同一 branch 的 run 数量增长，单文件模型会带来：
- 启动时 JSON 反序列化成本上升
- 每次写入都要全量重写，I/O 放大
- 文件体积增大后，人工排查可读性下降

### 12.2 建议演进方案：滚动归档

在保持现有 JSON 结构兼容的前提下，新增归档目录：
- `.multiverse/verses/archive/<sanitized-branch>/`

建议策略：
- 当 `runs.length` 超过阈值（例如 5,000 或 10,000）时触发归档
- 将较旧 runs 按时间段切分到归档文件（例如 `2026-04.json`）
- 主文件仅保留最近 N 条（例如 500 或 1,000）
- 主文件新增 `archivedSegments` 元信息，指向归档文件和时间范围

预期收益：
- 主路径读写性能稳定
- 历史数据可保留且可追溯
- 不影响 Story 1.2 当前交付范围

## 13. Corner Cases 补充清单

以下条目用于降低 1.2 落地后的隐性故障风险。标记为“1.2 内决策”的项，应在本 story 实施前明确规则。

### 13.1 1.2 内决策（建议本次就定）

1. 文件名碰撞（`feature/a` 与 `feature_a`）
- 风险：sanitize 后可能映射到同一文件。
- 建议：文件名采用 `sanitize(branch) + \"__\" + shortHash(branch)`。

2. 大小写不敏感文件系统碰撞（macOS/Windows）
- 风险：`Main` 与 `main` 可能冲突。
- 建议：文件名统一小写后再拼接 hash。

3. detached HEAD 聚合污染
- 风险：全部写到 `detached-head.json` 会混入不同 commit 的 run。
- 建议：detached 场景使用 `detached-<shortCommit>.json`。

4. run 启动记录时机
- 风险：在容器未真正启动前写入 start，可能产生语义不清的“未结束 run”。
- 建议：二选一并固定：
  - A: 容器启动成功后再写 `appendRunStart`
  - B: 保持先写 start，但增加状态字段区分 `start_failed/running/finished`

5. 锁文件陈旧处理
- 风险：崩溃后残留 `.lock` 造成持续超时。
- 建议：锁文件写入 `pid/host/createdAt`，检测持锁进程是否存活后安全回收。

6. 项目目录不可写
- 风险：无法创建 `.multiverse/verses`，启动流程中断。
- 建议：提供明确错误文案与操作建议（权限修复或切换可写目录）。

### 13.2 后续演进（可在后续 story 处理）

7. 损坏 JSON 的恢复路径
- 风险：当前“拒绝覆盖”虽然安全，但可用性较差。
- 建议：新增自动备份 `*.corrupted-<ts>.bak` + 可选最小重建策略。

8. 多工作树/子目录一致性
- 风险：不同启动目录可能映射不同 project root。
- 建议：统一 root 判定（推荐 git root）并文档化。

9. 容器 ID 缺失的 finalize 场景
- 风险：失败路径拿不到 `containerId`，但 run 仍需闭合。
- 建议：保持 `containerId` 可选，确保 `endAt/exitCode` 必填。

10. Schema 前向兼容
- 风险：后续引入 `archivedSegments` 时老 reader 兼容性不明确。
- 建议：在主文件中引入 `schemaVersion`（初始为 `1`）。
