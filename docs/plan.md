## 核心用户旅程

### 旅程 1：首次使用（最小可用路径）
**角色**：开发者 Alice  
**目标**：快速启动一个隔离的 coding agent 环境

**步骤**：
1. Alice 在项目目录执行 `multiverse start`
2. 系统检测到首次使用，使用默认配置创建 verse
3. 系统启动 Docker 容器，注入配置和凭证
4. Alice 进入 claude-code 交互模式，开始工作
5. Alice 退出后，系统自动保存运行记录

**价值**：零配置快速启动，立即可用

---

### 旅程 2：配置管理（核心价值）
**角色**：开发者 Bob  
**目标**：管理和复用不同的配置组合

**步骤**：
1. Bob 打开 GUI，编辑 CLAUDE.md 配置
2. Bob 添加自定义 skills
3. Bob 将当前配置保存为 template "我的开发环境"
4. Bob 在新项目执行 `multiverse start --template "我的开发环境"`
5. 系统使用保存的配置启动容器

**价值**：配置复用，提升效率

---

### 旅程 3：实验和对比（高级价值）
**角色**：研究员 Carol  
**目标**：对比不同配置的效果

**步骤**：
1. Carol 在 main 分支使用 template A 运行多次
2. Carol 创建新分支，使用 template B 运行多次
3. Carol 在 GUI 中查看两个 verse 的性能对比
4. Carol 根据数据选择更优的配置

**价值**：数据驱动决策

## 开发原则

### 1. 小步快走
- 每个 story 独立可测试
- 每个 sprint 交付可用功能
- 优先实现核心价值

### 2. 用户价值优先
- 标记 ⭐ 的 story 是核心价值
- 先实现用户旅程，再完善细节
- 每个功能都有明确的用户价值

### 3. 快速反馈
- 每个 sprint 结束 demo
- 及时收集用户反馈
- 根据反馈调整优先级

### 4. 技术债务管理
- 每个 sprint 预留 20% 时间重构
- 单元测试覆盖率 > 70%
- 代码 review 必须通过

---

## 风险管理

### 技术风险
1. **Docker 集成复杂度**：提前技术验证
2. **OTel 数据量**：实现采样机制
3. **Tauri 学习曲线**：提前学习

### 进度风险
1. **需求变更**：敏捷响应，调整优先级
2. **技术难题**：及时升级，寻求帮助
